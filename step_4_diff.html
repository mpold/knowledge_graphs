Step 4 — diff (diff_two.py)

Diffs two step-3 gene–gene graphs into one network: node colour is a continuous balance between the two sides, edge colour is the pair's provenance. Pure standard library. Nothing is hardcoded — the inputs, the output, the heading and both side labels are asked for at the prompt, so the script has no idea what it is diffing until it is run. This step should be viewed as separate from steps 1–3. A dedicated directory for this step is desirable.

The split that matters. The score is a slider, so the model cannot be baked at build time. Python ships the raw evidence — every sentence at or above the lowest step, plus degree tables for each step — and the browser recomputes which pairs qualify, each gene's connectivity and the balance index every time a control moves. Python decides what could ever be shown; the page decides what is shown now.

2. What qualifies as a drawn pair

Applied per score step. A pair is an unordered gene couple: A→B and B→A, and any positive/negated split of the same couple, all fold onto one key.

  1. Keep the pair's sentences scoring ≥ the active score. No sentences left in either graph → the pair does not exist at this step.
  2. The pair is shared if both graphs still have a sentence, otherwise it belongs to whichever graph does.
  3. Draw it if it is shared, or if either graph backs it with ≥ MIN_PUBS distinct publications (default 2).
Prominence is publications, not sentences. In these graphs the great majority of pairs rest on a single paper, so a sentence count mostly ranks how often one author repeated themselves rather than how well attested a relation is. A pair whose nine sentences all carry one PMID is one claim, not nine.
The rule is not symmetric, and it shows. Shared pairs bypass the publication test entirely, while an exclusive pair must clear it. Because exclusive claims in a smaller graph are especially likely to rest on a single paper, the rule culls the smaller side hardest — in one run it removed 97% of one graph's exclusive pairs against 3% of the other's, so a genuinely lopsided difference can be made to look like agreement. Raise or lower MIN_PUBS when the two inputs are very different in size.

Only the floor's selection ships. That is sufficient for every step above it: sentences passing the top step are a subset of those passing the floor, so a pair drawn at any step is always present at the floor. This is what keeps the payload to the pairs that could ever be drawn rather than every pair in both graphs.

3. The balance index (what makes a gene red or blue)

Computed in the browser at the active step, over the full graphs — never the subgraph on screen. Focusing a gene, brushing the palette or matching sentence text changes what is drawn, never what a gene is.

  1. Count partners. The gene's distinct neighbours, in each graph separately.
  2. Divide by each graph's own size. The two graphs are never the same size, so raw counts are not comparable. Each degree becomes a share of its own graph's total connectivity, where total = 2 × pairs (every pair has two endpoints).
  3. Contrast the two shares, bounded −1…+1.
  4. Name the number — colour uses the continuous value; the words are bucketed for readability only.
share_1 = degree_1 / total_1        (input 1 → the red pole)
share_2 = degree_2 / total_2        (input 2 → the blue pole)

index = (share_1 − share_2) / (share_1 + share_2)

  +1  pure input 1  ·  0  balanced  ·  −1  pure input 2

Dividing by the sum makes this a proportion rather than a magnitude: 3-vs-1 partners and 30-vs-10 read as the same lean. Magnitude is not lost — it is carried separately, by node size.

Why the normalisation is load-bearing

One literature has always been written about more than the other. On raw counts, a gene central to both is dragged toward whichever graph is larger and reads as though it belonged there. Normalising asks the same question of each graph — what fraction of this graph's wiring runs through this gene? — which is answerable on equal terms however lopsided the two inputs are.

Schematically, with input 2 the larger graph:

geneinput 1input 2 rawindexwhat it really is
A160+1.00 +1.00exclusive to input 1 — both agree
B2690−0.55 +0.13a hub central to both — raw calls it input 2's
C3563−0.29 +0.43leans to input 1 despite more raw input-2 partners
D0117−1.00 −1.00exclusive to input 2 — both agree
The two columns only agree at the poles. In between, raw reports the size difference between the two literatures rather than anything about the gene — rows B and C are exactly the genes a reader most wants to place, and exactly the ones it gets wrong.

The page's How balance works panel makes this argument live, on whatever two graphs were loaded: it derives its worked examples from the data rather than a fixed gene list, works out for itself which graph is larger, and counts the genes whose raw and normalised labels actually disagree.

4. Inputs

Everything comes from the prompt — there are no hardcoded paths and no defaults. A bare name resolves next to the script (os.path.dirname(__file__)); an absolute path is taken as given. All six are required; a name that is not an existing file simply re-prompts. Order matters: input 1 becomes the red pole of the scale, input 2 the blue pole.

promptneedfromwhat the script uses it for
input file name 1: required step 3 Its const DATA: edges[].from/.to, .cat (positive / negated) and .sents[] (pmid, text, sc, yr). Becomes the red side of every pair.
input file name 2: required step 3 The same fields, for the blue side — and the vis-network bundle, lifted out of its first <script> and inlined into the output so the result opens offline.
output file name: required you Where to write. A bare name gets .html appended; naming either input re-prompts.
graph name: required you The page's heading and <title>, HTML-escaped on the way in.
label 1: required you What to call input 1 everywhere — legend, tooltips, table headers, selection panel, the balance buckets (“pure X”, “X-leaning”) and the explainer's prose and formulae.
label 2: required you The same, for input 2.

Node metadata in the sources (drug targets, chemical categories, node colours) is deliberately ignored: this graph re-derives colour from the diff, so the sources' own palettes would only conflict. Gene identity is taken as given — symbols arrive already normalised to HGNC by step 3, which is why the page tells the reader to expect the nodes and the sentences to use different vocabulary for the same molecule.

Encodings in the output

Provenance and balance are measured on the full evidence at the active score, never on the filtered subset. A text query that happens to hit only one side's sentences must not repaint a shared pair as one-sided — that would be a false claim about the biology. The filters select which pairs to show; they never restate what they are.
Node colour and edge colour answer different questions, and will disagree. Colour is a property of the gene across both graphs; provenance is a property of one pair. A blue gene having a red edge is not a bug — it means a gene wired mostly in input 2 has a relation seen only in input 1, which is usually the interesting case.

6. Options

There is no command line. File names, the heading and both labels come from the prompt; the thresholds and palette are constants at the top of the script; everything else is a control in the finished page.

Build-time constants

constantdefaulteffect
SCORE_STEPS0.75, 0.80, 0.85, 0.90, 0.95…0.99 The slider's stops — coarse 0.05s through the noisy low band, 0.01s from 0.95 up where a single point still changes the graph. Its minimum is the payload floor.

RAMP_MODESblue↔red Diverging ramp anchors per light/dark mode, plus the outline lightness shift.

Score, cluster size, palette range, gene focus, hops, text query and the two legend states persist in localStorage under the diff-two. prefix — every access wrapped in try/catch, because file:// origins may refuse storage outright and a saved preference must never take the graph down with it. The score is stored as its value, not the slider index, so a later edit to SCORE_STEPS cannot quietly restore someone to a different threshold.

7. How to run

  1. 1python diff_two.py — then answer six prompts. No arguments, no network, no dependencies. 
    input file name 1: first_G.html
input file name 2: second_G.html
output file name:  diff.html
graph name:        First vs Second
label 1:           First
label 2:           Second
  2. 2Scripted, by piping the six answers in order: 
    printf '%s\n' first_G.html second_G.html diff.html "First vs Second" First Second | python diff_two.py
  3. 3To diff any other pair, answer the prompts with their graphs and labels. Nothing in the script is specific to a disease or a file — not the paths, not the title, not the wording, and not the explainer's worked examples.
Run this after step 3 has produced both _G.html graphs. The two must come from the same pipeline version — the diff assumes both carry the same DATA shape and the same HGNC-normalised symbols, and it compares them without re-checking either.

8. Notes

  • Nothing is disease-specific. Both sides are named at the prompt and those labels flow through every string in the page. The explainer goes further and is derived: its worked examples are picked from the data (the best-connected genes, walked across the scale) rather than a hardcoded gene list. A fixed gene list would be wrong for every pair but the one it was written for.
  • The prompt refuses to overwrite an input. Naming either source graph re-prompts rather than writing — the script reads those files, and a typo would destroy megabytes of upstream pipeline output irrecoverably. An empty answer is rejected too; EOF aborts cleanly.
  • Deterministic. Identical inputs and answers give a byte-identical output; there is no timestamp, no randomness and no ordering that depends on dict iteration.
  • The colour ramp is computed, not chosen. The red arm is generated to mirror the blue ramp's OKLCH lightness, and both arms are monotonic in lightness — the check a diverging ramp answers to. Blue↔red survives every simulated colour-vision deficiency (worst case protanopia, ΔE 15.1 against a ≥ 8 target); a blue↔green scale would not.
  • Two rounding traps lurk in the OKLab port, and both would silently shift colours: JavaScript's Math.round breaks .5 upward while Python's round() breaks to even (hence floor(x + 0.5)), and Math.cbrt accepts negatives while ** (1/3) raises (hence copysign).
  • Expect the two graphs to overlap less than you think. In one run the inputs shared 42 pairs out of 550 and 2,560; of the exclusive pairs whose both genes existed in the other graph, only 17% were an edge there. The literatures tend to connect the same genes differently rather than discuss different genes — which is the whole reason to diff them.
  • The low end of the score is noisier by construction. The floor admits relations the extractor was only ~three-quarters confident about, which is why the fine steps live up top where precision matters.
