Step 1 — publications pipeline

Order of execution, inputs and outputs of the Python scripts in the lung/ directory (top level only — the gpu_bundle/ Kaggle pipeline is covered separately). The seven scripts form a single linear pipeline that turns one PubMed query into a de-duplicated, full-text corpus ready for named-entity recognition.

Run the scripts in the order below: each consumes the files the previous one wrote, and writes an HTML summary of its own stage into summaries/ (each summary is named after its script). Every script creates the input and output directories it needs if they are missing, and (except grobid_xml.py) empties its bulk output directory before writing so each run starts clean — Python scripts are never deleted. Several stages reach the network (NCBI E-utilities, OpenAlex, CrossRef) or need Docker (GROBID), so they are not run automatically — run each yourself, or drive all seven in order with step_1_orchestrator.py (see section 3), which still needs the same network and Docker access.

1. Order of execution (data flow)

1. pubmed_query.py
in: PubMed query (STDIN)  →  out: pmids/pmid_pmc_ids.tsv
↓  pmid_pmc_ids.tsv
2. high_impact_xml.py
in: pmids/pmid_pmc_ids.tsv  →  out: high_impact_xmls/PMC*.xml
↓  high_impact_xmls/
3. xml_structure.py
in: high_impact_xmls/PMC*.xml  →  out: summaries/xml_structure.html (flags the no-<body> subset)
↓  the no-<body> records
4. ncbi_pdf.py
in: no-<body> high_impact_xmls/PMC*.xml  →  out: ncbi_pdfs_grobid/PMC*.pdf
↓  ncbi_pdfs_grobid/
5. grobid_xml.py needs Docker + GROBID
in: ncbi_pdfs_grobid/PMC*.pdf  →  out: grobid_xmls/PMC*.grobid.tei.xml
↓  grobid_xmls/ + high_impact_xmls/
6. named_entity_xml.py
in: grobid_xmls/ + high_impact_xmls/  →  out: named_entity_xmls/, gpu_bundle/experimental_ner/
↓  gpu_bundle/experimental_ner/
7. pre_ner_xml_structure.py
in: gpu_bundle/experimental_ner/  →  out: summaries/pre_ner_xml_structure.html

Step 3 (xml_structure.py) is an analysis/reporting step: it does not transform files but identifies the no-<body> subset of high_impact_xmls/ that step 4 targets, so it is placed before ncbi_pdf.py. (ncbi_pdf.py also re-detects that subset itself, so it does not strictly depend on step 3 having run.) Step 5 (grobid_xml.py) is the only stage that keeps resume (it skips PDFs that already have TEI) rather than wiping its output dir; it is also skipped entirely when every article already has JATS full text.

3. How to run

One command — the orchestrator

step_1_orchestrator.py runs all seven scripts in the order above, in one process. It feeds the PubMed query to step 1 on STDIN and aborts the moment any stage exits non-zero (so a failed stage never silently poisons the next). Every path is script-relative, so it can be launched from any working directory.

# query as an argument (fed to step 1's STDIN)
python step_1_orchestrator.py "your pubmed query"

# or pipe it in / type it at the prompt
echo "your pubmed query" | python step_1_orchestrator.py
python step_1_orchestrator.py

Options: --start N / --stop N / --only N run a sub-range of steps 1–7 (e.g. --only 5 to re-run just GROBID, or --start 4 to resume after a manual step 3); --list prints the pipeline order; --dry-run shows what would run without executing anything. The query is only read when step 1 is in the selected range. Environment variables (NCBI_API_KEY, TIME_BUDGET, GROBID settings, …) are passed straight through to each child, so the orchestrator changes nothing about the requirements below — the network stages still reach out and step 5 still needs Docker + GROBID.

Or run each stage by hand

  1. 1echo "your query" | python pubmed_query.py — or run it and type the query at the prompt.
  2. 2python high_impact_xml.py
  3. 3python xml_structure.py
  4. 4python ncbi_pdf.py
  5. 5python grobid_xml.py needs Docker + GROBID
  6. 6python named_entity_xml.py
  7. 7python pre_ner_xml_structure.py
These scripts are not executed automatically: most reach external services (NCBI E-utilities, OpenAlex, CrossRef) and step 5 needs a running GROBID container. Set NCBI_API_KEY to lift the 3 req/s rate limit. Each run starts clean (the bulk output dir is emptied before writing) except grobid_xml.py, which resumes; bound the network stages with TIME_BUDGET if needed.

4. Notes

  • Single linear pipeline. The output directory of each step is the input of the next: pmids/ → high_impact_xmls/ → ncbi_pdfs_grobid/ → grobid_xmls/ → named_entity_xmls/ → gpu_bundle/experimental_ner/.
  • All paths are script-relative. Each script resolves inputs/outputs from its own location, so it can be run from any working directory (and a copied tree runs in isolation).
  • Directories are auto-created. Every script creates the input and output directories it needs if they are missing; file-based inputs (e.g. pmid_pmc_ids.tsv) must still be produced by the previous stage.
  • Seven self-contained HTML summaries land in summaries/, each named after its script (pubmed_query.html, high_impact_xml.html, xml_structure.html, ncbi_pdf.html, grobid_xml.html, named_entity_xml.html, pre_ner_xml_structure.html).
  • Hand-off to step 2. The assembled NER corpus in gpu_bundle/experimental_ner/ is the input to the GPU bundle (gpu_bundle/gpu.py); see step 2.
  • Orchestrator, not a stage. step_1_orchestrator.py only chains the seven scripts in order (feeding the query to step 1 on STDIN, aborting on the first non-zero exit); it produces no corpus or summary of its own and can be run partially with --start/--stop/--only. It does not relax any per-stage requirement — the network stages and step 5's GROBID container are still needed.
