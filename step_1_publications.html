Step 1 — publications pipeline

Order of execution, inputs and outputs of the Python scripts in the lung/ directory (top level only — the gpu_bundle/ Kaggle pipeline is covered separately). The seven scripts form a single linear pipeline that turns one PubMed query into a de-duplicated, full-text corpus ready for named-entity recognition.

linear
each step feeds the next
Run the scripts in the order below: each consumes the files the previous one wrote, and writes an HTML summary of its own stage into summaries/ (each summary is named after its script). Every script creates the input and output directories it needs if they are missing, and (except grobid_xml.py) empties its bulk output directory before writing so each run starts clean — Python scripts are never deleted. Several stages reach the network (NCBI E-utilities, OpenAlex, CrossRef) or need Docker (GROBID), so they are not run automatically — run each yourself, or drive all seven in order with step_1_orchestrator.py (see section 3), which still needs the same network and Docker access.

Step 3 (xml_structure.py) is an analysis/reporting step: it does not transform files but identifies the no-<body> subset of high_impact_xmls/ that step 4 targets, so it is placed before ncbi_pdf.py. (ncbi_pdf.py also re-detects that subset itself, so it does not strictly depend on step 3 having run.) Step 5 (grobid_xml.py) is the only stage that keeps resume (it skips PDFs that already have TEI) rather than wiping its output dir; it is also skipped entirely when every article already has JATS full text.

Typical use: strip from this project's gpu_bundle/experimental_ner/ any files that also appear in another corpus. Not part of the linear pipeline and not run by the orchestrator.

3. How to run

One command — the orchestrator

step_1_orchestrator.py runs all seven scripts in the order above, in one process. It collects two inputs on STDIN up front — the PubMed query for step 1, then the publication impact percentile for step 2 — and feeds each to the stage that consumes it. It aborts the moment any stage exits non-zero (so a failed stage never silently poisons the next). Every path is script-relative, so it can be launched from any working directory.

# query as an argument (fed to step 1's STDIN);
# the impact percentile is still prompted on its own line
python step_1_orchestrator.py "your pubmed query"
Publication impact percentile (decimal between 0 and 1): 0.90

# or pipe both lines in (query first, percentile second)
printf 'your pubmed query\n0.90\n' | python step_1_orchestrator.py

# or type each at its prompt
python step_1_orchestrator.py

The orchestrator prompts for the percentile on its own dedicated line after the query and hands it to high_impact_xml.py on that script's STDIN. Enter a decimal between 0 and 1 (e.g. 0.90 = top 10% by article-weighted impact factor); a blank line (or EOF) falls back to PERCENTILE=0.90, and IF_THRESHOLD overrides the percentile entirely.

Options: --start N / --stop N / --only N run a sub-range of steps 1–7 (e.g. --only 5 to re-run just GROBID, or --start 4 to resume after a manual step 3); --list prints the pipeline order; --dry-run shows what would run without executing anything. The query is only read when step 1 is in the selected range, and the percentile only when step 2 is. Environment variables (NCBI_API_KEY, TIME_BUDGET, GROBID settings, …) are passed straight through to each child, so the orchestrator changes nothing about the requirements below — the network stages still reach out and step 5 still needs Docker + GROBID.

Or run each stage by hand

  1. 1echo "your query" | python pubmed_query.py — or run it and type the query at the prompt.
  2. 2echo 0.90 | python high_impact_xml.py — or run it and type the impact percentile (decimal 0–1) at the prompt; blank falls back to PERCENTILE=0.90.
  3. 3python xml_structure.py
  4. 4python ncbi_pdf.py
  5. 5python grobid_xml.py needs Docker + GROBID
  6. 6python named_entity_xml.py
  7. 7python pre_ner_xml_structure.py

Optional — subtract.py optional: not one of the seven and not chained by the orchestrator. Run it by hand to remove from one directory the entries whose names also appear in another. It reads the two directory paths on STDIN (directory_1 first, directory_2 second), moves the overlap into gpu_bundle/removed/, and writes summaries/subtract_optional.html:

# move gpu_bundle/experimental_ner entries whose names also exist in another corpus aside
printf 'gpu_bundle/experimental_ner\n/path/to/other/gpu_bundle/experimental_ner\n' | python subtract.py

# or run it and type the two directory paths at the prompts
python subtract.py
These scripts are not executed automatically: most reach external services (NCBI E-utilities, OpenAlex, CrossRef) and step 5 needs a running GROBID container. Set NCBI_API_KEY to lift the 3 req/s rate limit. Each run starts clean (the bulk output dir is emptied before writing) except grobid_xml.py, which resumes; bound the network stages with TIME_BUDGET if needed.

4. Notes

  • Single linear pipeline. The output directory of each step is the input of the next: pmids/ → high_impact_xmls/ → ncbi_pdfs_grobid/ → grobid_xmls/ → named_entity_xmls/ → gpu_bundle/experimental_ner/.
  • All paths are script-relative. Each script resolves inputs/outputs from its own location, so it can be run from any working directory (and a copied tree runs in isolation).
  • Directories are auto-created. Every script creates the input and output directories it needs if they are missing; file-based inputs (e.g. pmid_pmc_ids.tsv) must still be produced by the previous stage.
  • Seven self-contained HTML summaries land in summaries/, each named after its script (pubmed_query.html, high_impact_xml.html, xml_structure.html, ncbi_pdf.html, grobid_xml.html, named_entity_xml.html, pre_ner_xml_structure.html).
  • Hand-off to step 2. The assembled NER corpus in gpu_bundle/experimental_ner/ is the input to the GPU bundle (gpu_bundle/gpu.py); see step 2.
  • Orchestrator, not a stage. step_1_orchestrator.py only chains the seven scripts in order (feeding the query to step 1 and the impact percentile to step 2 on STDIN, aborting on the first non-zero exit); it produces no corpus or summary of its own and can be run partially with --start/--stop/--only. It does not relax any per-stage requirement — the network stages and step 5's GROBID container are still needed.
  • Optional cleanup utility. subtract.py is not one of the seven and is not run by the orchestrator: run it by hand when needed. It moves entries of directory_1 whose name also appears in directory_2 into gpu_bundle/removed/ (relocated, not deleted) and writes an eighth, conditional summary summaries/subtract_optional.html — useful for de-duplicating this project's gpu_bundle/experimental_ner/ against another corpus.
