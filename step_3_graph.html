Step 3 — graph (high_confidence.py & high_confidence_g.py)

NB!  kaggle_working.zip must be downloaded to the local computer and unzipped before high_confidence.py or high_confidence_g.py is triggered.

Inputs and outputs of the two step-3 analysis scripts that read the triples the GPU bundle (step 2) produced and draw the interactive brain-cancer gene–gene graph. They are standalone scripts (not orchestrated by gpu.py) — one command in, one JSON export plus one self-contained HTML graph out — and differ in exactly one place, the qualifying filter: high_confidence.py (sections 1–7 below) keeps the high-confidence gene–gene relations in a disease/chemical context (the “G_D_C” filter), while high_confidence_g.py (section 8) drops that context requirement and keeps every high-confidence gene–gene relation (the gene-only “G” filter). Their _G-suffixed outputs never clobber each other, so both can target the same data root.

2. The “G_D_C” qualifying filter

A scored RE triple is kept only when all five conditions hold (qualifies()). This is what separates “high-confidence gene–gene in a disease/chemical context” from the raw scored output.

  1. score ≥ --score — model relation score at or above the threshold (export default 0.8).
  2. control = no — at least one GENETIC endpoint is annotated control: "no" (a studied gene, not a lab control).
  3. not control = yes — neither endpoint is annotated control: "yes".
  4. no MKI67 — neither endpoint's hgnc_symbol is MKI67 (proliferation marker, excluded).
  5. disease/chemical context — the triple's sentence also carries a DISEASE or CHEMICAL entity, per triples.json.

6. How to run

  1. 1python high_confidence.py — reads kaggle_working/, writes the JSON + graph at the defaults (score 0.8).
  2. 2python high_confidence.py --data-root path/to/run --score 0.95 — point at a different bundle output tree and raise the export threshold.
  3. 3python high_confidence.py --no-graph — JSON export only (no network, no HTML).
Run this after the GPU bundle (step 2): it needs at least TRIPLES/triples_re_GENETIC_DISEASE_CHEMICAL_normalized.json and TRIPLES/triples.json under the data root, or it exits with an error. The optional inputs (years, chemical targets, MONDO labels) just enrich the graph's filters; the graph still renders without them. Building the graph fetches vis-network from the CDN — harmless if offline (it falls back to a CDN link, or use --no-graph).

7. Notes

  • Hand-off from step 2. Every required input is a file the GPU bundle (step_2_triples.html) wrote into TRIPLES/; the optional inputs come from the bundle's DISEASE/, CHEMICAL/ and databases/ dirs. This stage is the “step 3” that consumes them.
  • Export threshold vs. graph universe. --score sets the exported JSON's cutoff, but the graph is always built from the score ≥ 0.8 universe (with per-sentence scores kept), so the ≥0.8 / ≥0.95 / ≥0.99 toggle works in the browser regardless of --score.
  • Single-file, portable HTML. The graph inlines both the data payload and (when the fetch succeeds) the vis-network library, so high_confidence.html opens with no build step and no network once written.

8. Gene-only variant — high_confidence_g.py (the “G” step)

A sibling script, high_confidence_g.py, runs the same pipeline as a separate step with one logic difference: it applies the gene-only “G” filter instead of the “G_D_C” filter above. Everything else — inputs, graph rendering, options, the in-browser confidence toggle — is identical. Its output filenames all differ (a _G suffix), so the two scripts can be run against the same data root without clobbering each other.

The one difference: high_confidence_g.py drops qualifying rule 5 — the requirement that the triple's sentence also carry a DISEASE or CHEMICAL entity. A triple qualifies on the first four rules alone (score, control = no, not control = yes, no MKI67). This yields a larger universe: every high-confidence gene–gene relation, whether or not its sentence is in a disease/chemical context.

The “G” qualifying filter

A scored RE triple is kept when all four conditions hold (qualifies() in high_confidence_g.py) — rules 1–4 of the G_D_C filter, without rule 5:

  1. score ≥ --score — model relation score at or above the threshold (export default 0.8).
  2. control = no — at least one GENETIC endpoint is annotated control: "no".
  3. not control = yes — neither endpoint is annotated control: "yes".
  4. no MKI67 — neither endpoint's hgnc_symbol is MKI67.
  5. disease/chemical context — not applied in this variant (this is the only difference from the G_D_C filter).

Inputs

Identical to the G_D_C script (section 3): the same two required files and the same six optional graph inputs, all under --data-root. Note TRIPLES/triples.json is still required here, but it is used only for the graph's per-sentence disease labels — not for filtering (rule 5 is gone).

Outputs (all _G-suffixed, distinct from the G_D_C script)
